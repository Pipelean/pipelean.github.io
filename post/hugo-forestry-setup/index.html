Hugo website, Aerobatic CDN, Forestry.io web dashboard and Git to bind them all [Tutorial]

   2017 Jan 26      technology      cms · static · serverless · hugo · tutorial

A brief tutorial to create a completely static website with a template engine, deployed to a CDN and with an admin dashboard for your content editors.

Prerequisites

You are a programmer and know how to build websites and understand the difference between a static site, a template generator for static site and a server that generates HTML pages dynamically from templates and data. You’re a little familiar with Jekyll or another static site generator: you know what they are and what they do, not necessarily already used once. You know Git and how to use it.

You’re not afraid of change.

Why and How

Is beyond the scope of this article and in general beyond the scopeto both convince anyone about how good is life with a static site generator in place of a Wordpress (or Drupal, Joomla, you name it…) or any PHP+MySQL+Server solution.

There is, in fact, this article I wrote on why build a CMS solution without a server and how. Step by step

I will proceed with a step by step walktrough. If you’d like to read about the general solution before for an high level overview of how this works, the aforementioned article is what you need now.

Static site with Hugo

Is beyond the scope of this article to explain how a Hugo website is built. It’s not something you learn in 5 minutes but is pretty easy for the avarage programmer and one can be generated in 2 minutes. That, or just grab a sample one and move on.

Git on Bitbucket

Let’s create a new private (and free) Git repository on bitbucket. Let’s also create and push a new empty branch called ‘hosting’ (or whatever you like, but in this article we’ll refer to this branch with that label).

Now you can ‘git init’ your hugo project locally and push it to master. Not to hosting, just to master.

Setup the CDN: Aerobatic

Now is time to setup Aerobatic, which will be the CDN where we publish our website. Aerobatic is a standalone product, but is seamlessy integrated with bitbucket, and this is very convenient.

One: go to Aerobatic Settings in Bitbucket admin dashboard, create a new site with a new name. I will name it “hugosample” in this article. The the public URL will be hugosample.aerobatic.io and is now empty. Good.

Two: in Settings -> deploy branches: https://bitbucket.org/pipeleancom/pipeleancom/addon/aerobatic-bitbucket-addon/aerobatic-app-dashboard with the wrong branch.

Now aerobatic is ready and will show on http://hugosample.aerobatic.io whatever you have on the hosting branch. Now you do not have anything cause someone have to build your hugo website like you do locally from the command line. Who’s going to do it?

Write some config (one line)

Ok, now we have almost all the pieces in place.

Hugo sites has a main cofiguration file, called config.toml, in the project root. Open it, edit the baseURL property, which is normally the first line in the file, to something like:

baseurl = “https://samplesite.aerobatic.io/"

Important: The final slash MUST be there, or your resources will not load properly.
I know. I hate it too. We can live with that.

Build Hugo in the cloud

Forestry will both build your Hugo project from master to hosting.

Signin to Forestry.io and create a new site. You’ll be asked for a repository and a hosting site. Pick bitbucket for both. Use “master” branch as repository and “hosting” branch for hosting.

Try it: the web dashboard

Forestry also provides a Web Dashboard to administer the site and for the content editors to do their job.

Let’s open the project, create a new content and write something.

Click preview. Does the preview works? Good. Now publish it. There is very good chance that Forestry has succesfully built the Hugo project and pushed the result to the Git repository ‘hosting’ branch which is finally published on your site. Go to your like http://samplehugo.aerobatic.io.

Do you see your site with your new content? Great!

You’re all set. The good guys have won.

Troubleshooting

Doesn’t work? I know, it does never work the first time. Here are a few things that could have gone wrong.

  • If the Hugo theme is a git submodule, you MUST have the .gitmodules file or Forestry won’t be able to checkout the whole code and so can’t built Hugo site.
  • in config.toml, baseUrl MUST contain the final slash!
  • is baseURL correct? Really?

Still doesn’t work? Ok, leave me a tweet

Bonus tip

use bitbucket branch permission feature and allow no one to write on “hosting” and only trusted people to write on master. Forestry will continue to work as expected, you do not need to explicitely add a user for forestry inside bitbucket permission settings.
